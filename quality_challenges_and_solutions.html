Quality: challenges and solutions (2017)

There are a number of things that can stand in the way of good quality. Here are some of them, and the solutions we have in place for dealing with them.

Challenge

Example solutions

Distance from the problems

  • Have everyone working from the same base branch. If something breaks there's a good chance of it getting noticed by a colleague.

  • Know your game: Many team members playing the game actively as players.

  • Testing Dojos: Regular testing sessions with a sub-team or the entire team for bigger updates (organized by the sub-team).

  • Mindset for quality: The full sub-team is responsible for the quality, because they ship it. Goes hand in hand with the QA process (QA Assistance → Developers test and demo their work), which only works if people care for quality.

"Broken windows"

  • Stable main branch policy. Makes it very clear what needs to be done when there's a bug (fix it right now! it's affecting everyone!), and also makes it easier to spot real issues in the first place.

  • Constantly fixing bugs, no "bug fixing day". Quality should be a constant, not something to be procrastinated.

  • High priority for bugs helps to maintain a low level of bugs. Note: Maintaining a certain level of quality is fairly straightforward once you've gotten there in the first place.

Stress

  • Sprint goals shouldn't rely on 100% team utilization. Extra time for iteration, bug-fixing, and moving quickly.

  • Daily beta releases: Have things go live on Beta before the release train departs so that there's time for them to be fixed.

  • Pioneer market: We release the browser version in a single market first to catch bugs before worldwide rollout.

  • Pioneer platform: We release the mobile version to a percentage of our players using Google Play a day before we release everything else.

Costly delays between work finishing and finding issues

  • Have automated tests (functional tests & UI checks) running on pull requests and nightly builds.

  • Release to Beta early and often to get feedback and bug reports in time to still fix them easily.

Issue visibility

  • Make the current bug count transparent (in stand-ups, component syncs, retros etc.), and celebrate improvements.

  • Release Management Overview: Clear prioritization of bugs that need dealing with first, and bugs that require fixing before release, or fixes that need placing into the release branch are clearly marked and tracked.

  • People know where to find bugs to work on (one JIRA filter that everyone uses).

Miscommunication

  • CM+GD+PM collaboration for announcements.
